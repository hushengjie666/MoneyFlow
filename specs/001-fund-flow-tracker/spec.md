# Feature Specification: 资金动向登记与实时余额看板

**Feature Branch**: `[001-fund-flow-tracker]`  
**Created**: 2026-03-03  
**Status**: Draft  
**Input**: User description: "构建一个应用程序，支持我登记资金动向，比如登记周期性或一次性的资金入项或者资金出项，系统主页可以每分每秒实时显示资金金额跳动，跳动的数据来源就源自周期性或一次性的资金事件，其次可以支持初始化存款，以及显示最新存款数据。"

## Clarifications

### Session 2026-03-03

- Q: 周期事件的记账方式采用哪种规则？ -> A: A（周期事件按周期总额线性分摊到每一秒，余额连续变化）
- Q: 用户模型采用哪种范围？ -> A: A（单用户本地账本，无多账户协作）
- Q: 余额是否允许为负？ -> A: A（允许负余额，可透支显示）
- Q: 时间基准采用哪种时区策略？ -> A: B（按用户本地时区存储与计算，当前默认 Asia/Shanghai）
- Q: 金额精度采用哪种内部计算方式？ -> A: A（金额输入与存储统一使用“元”，支持最多 2 位小数）
- Q: 事件“删除”语义如何定义？ -> A: A（采用软删除：状态置为 `deleted`，保留可追溯记录，不做物理删除）
- Q: 单用户模型下请求携带 `userId/accountId` 字段如何处理？ -> A: A（视为非法输入并拒绝写入）
- Q: 金额越界阈值如何定义？ -> A: A（事件金额 `0.01..1000000000` 元；初始化存款 `-1000000000..1000000000` 元）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 初始化并查看当前存款 (Priority: P1)

作为用户，我可以先设置一笔初始化存款，并在主页随时看到当前最新存款金额。

**Why this priority**: 没有初始化存款就无法建立可追踪的资金基线，后续所有资金变动都失去参照。  
**Independent Test**: 在空数据状态下设置初始化存款后，主页应立即显示该金额，并作为后续计算基数。

**Acceptance Scenarios**:

1. **Given** 系统首次使用且没有存款基线，**When** 用户录入初始化存款并保存，**Then** 系统保存该基线并在主页显示最新存款金额。
2. **Given** 系统已有初始化存款，**When** 用户刷新主页，**Then** 主页显示与已保存数据一致的最新存款金额。

---

### User Story 2 - 登记一次性资金入项/出项 (Priority: P1)

作为用户，我可以登记一次性的资金入项或资金出项，系统会即时更新可见余额，并参与后续实时跳动计算。

**Why this priority**: 一次性交易是最核心、最常见的资金动向记录场景。  
**Independent Test**: 录入一笔一次性入项或出项后，不依赖周期事件也能正确变更余额。

**Acceptance Scenarios**:

1. **Given** 当前余额为 X，**When** 用户新增一次性入项金额 A，**Then** 最新余额变为 X + A，且该事件记录可追溯。
2. **Given** 当前余额为 X，**When** 用户新增一次性出项金额 B，**Then** 最新余额变为 X - B，且该事件记录可追溯。

---

### User Story 3 - 登记周期性资金事件并驱动实时跳动 (Priority: P1)

作为用户，我可以登记周期性资金入项或出项，系统根据事件频率自动分摊并按秒更新主页金额，使余额持续跳动。

**Why this priority**: 你的核心需求是“每分每秒实时显示资金金额跳动”，而跳动来源就是周期性或一次性资金事件。  
**Independent Test**: 配置至少一条周期性事件后，主页无需手动刷新即可观察到金额按时间连续变化。

**Acceptance Scenarios**:

1. **Given** 用户创建了周期性入项事件（例如每月工资），**When** 主页处于打开状态，**Then** 系统按统一规则将周期金额分摊到秒级并持续更新显示余额。
2. **Given** 用户创建了周期性出项事件（例如每月房租），**When** 主页处于打开状态，**Then** 系统按统一规则将周期金额分摊到秒级并持续更新显示余额。
3. **Given** 周期性事件被暂停或删除，**When** 变更生效后进入下一秒，**Then** 余额跳动应按新事件集合重新计算。

---

### User Story 4 - 查看资金事件与最新存款数据 (Priority: P2)

作为用户，我希望查看最新存款值和近期资金事件，确认余额跳动来源是否正确。

**Why this priority**: 可解释性可降低对“余额为什么变化”的疑问，提升使用信任。  
**Independent Test**: 查看主页或事件列表，能够将余额变化与对应入项/出项事件关联。

**Acceptance Scenarios**:

1. **Given** 系统存在多条一次性与周期性事件，**When** 用户查看主页或事件区域，**Then** 可以看到最新存款值和最近事件摘要（类型、金额、时间/周期）。

---

### Edge Cases

- 初始化存款为 0 时，系统应允许保存并从 0 开始跳动计算。  
- 资金出项累计超过入项时，系统应继续计算并显示负余额。  
- 用户录入负数金额、空金额或非数字金额时，系统应阻止保存并提示可修正错误。  
- 周期事件开始时间晚于当前时间时，未生效前不应影响当前余额跳动。  
- 一次性事件时间戳位于未来时，不应提前计入当前余额。  
- 周期事件频率发生变更时，系统应从变更生效时刻开始使用新频率计算。  
- 当系统无任何周期事件时，余额应保持静态，仅在新增一次性事件时变更。  

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持用户设置并保存初始化存款，且初始化存款作为余额计算的起点。  
- **FR-002**: 系统 MUST 支持登记一次性资金事件，事件类型至少包含“入项”和“出项”。  
- **FR-003**: 系统 MUST 支持登记周期性资金事件，事件类型至少包含“入项”和“出项”，并支持定义周期频率（如日/周/月）。  
- **FR-004**: 系统 MUST 在主页持续显示“最新存款金额”，该金额由初始化存款与所有已生效资金事件共同计算得出。  
- **FR-005**: 系统 MUST 提供实时更新机制，使主页金额至少每秒刷新一次，呈现连续跳动效果。  
- **FR-006**: 系统 MUST 将周期性资金事件按周期总额线性分摊到秒级时间片后参与实时余额计算。  
- **FR-007**: 系统 MUST 在新增、编辑、删除（软删除）、暂停资金事件后，于下一个刷新周期内反映到主页金额。  
- **FR-008**: 系统 MUST 保留资金事件的最小可追溯信息：标题、事件类型（入/出）、金额、事件时间或周期定义、状态（启用/暂停）。  
- **FR-009**: 用户 MUST 能在主页查看最新存款数据，并查看能够解释余额变化的近期事件摘要。  
- **FR-010**: 系统 MUST 对非法输入进行校验（空值、非数值、金额越界），阻止错误数据写入；其中事件金额 MUST 在 `0.01..1000000000` 元区间内，初始化存款 MUST 在 `-1000000000..1000000000` 元区间内，金额最多保留 2 位小数。  
- **FR-011**: 系统 MUST 以单用户模型运行；当前版本不包含多用户隔离、协作和多账户管理能力；请求中出现 `userId/accountId` 等多账户字段时 MUST 返回校验错误。  
- **FR-012**: 系统 MUST 允许最新存款为负数，并在显示与计算链路中保持一致处理。  
- **FR-013**: 系统 MUST 按用户本地时区进行资金事件存储与余额计算；当前默认时区为 Asia/Shanghai。  
- **FR-014**: 系统 MUST 以“元”为业务金额单位，在输入、存储与接口层保持一致，并保证金额最多 2 位小数。  

### 资金数字跳动规则（重点约束，禁止随意改动）

以下规则是首页与悬浮组件共享的资金跳动基线，任何变更都必须先更新本节并补充对应回归测试。

1. 统一入口：
   - 展示层必须通过 `resolveJumpDisplayDeltaByUnit` 计算“每秒/每分/每时/每天/每周/每月/每年”显示值。
   - 主页面与悬浮组件必须使用同一计算模块，不允许复制并分叉公式。
2. 生效前提：
   - 仅统计 `eventKind=recurring` 且 `status=active` 且 `effectiveAt <= now` 且金额/间隔合法（`amountYuan>0`、`recurrenceInterval>=1`）的事件。
3. `second` / `minute` / `hour` 维度：
   - 必须同时满足：当天在 `activeWeekdays` 内，且当前时间在 `dailyStartTime <= now < dailyEndTime` 时间窗内。
   - 任一条件不满足时，该事件对 `second/minute/hour` 的贡献为 `0`。
   - 单事件公式：  
     `activeDaysPerPeriod = periodDays * (activeWeekdaysCount / 7)`  
     `activeHoursPerDay = (dailyEndTime - dailyStartTime)`  
     `hourDelta = amount / (activeDaysPerPeriod * activeHoursPerDay)`  
     `minuteDelta = hourDelta / 60`  
     `secondDelta = minuteDelta / 60`
4. `day` 维度：
   - 只受 `activeWeekdays` 影响，不受 `dailyStartTime/dailyEndTime` 影响（即使当前在时间窗外也可显示当天维度跳动）。
   - 单事件公式：`dayDelta = amount / activeDaysPerPeriod`。
5. `week` / `month` / `year` 维度：
   - 使用“名义周期流速”（`amount / recurrencePeriodSeconds`）换算，不按工作时段折算。
   - 单事件换算：`week = perSecond * 7d`，`month = perSecond * 30d`，`year = perSecond * 360d`。
6. 方向与叠加：
   - 入项为正、出项为负；多事件按代数和叠加。
7. 变更守则（强制）：
   - 修改任一跳动口径时，必须同步更新：
     - `frontend/src/jump-flow.js` 的对应实现
     - `tests/unit/jump-flow.test.js` 的回归样例
     - 本规范章节
   - 未同时完成上述三项，不得合入。

### Non-Functional Requirements *(mandatory)*

- **NFR-001**: 系统 MUST 为关键用户流定义一致体验，包括首次无数据、加载中、输入错误、保存成功、保存失败等状态。  
- **NFR-002**: 在单用户常规使用场景下，主页余额数值刷新间隔 MUST 不大于 1 秒；资金事件变更到余额可见更新的延迟 SHOULD 小于 2 秒。  
- **NFR-003**: 金额展示 MUST 保持格式一致（货币符号、千分位、小数位规则一致），并保证在常见桌面与移动端视口下清晰可读。  
- **NFR-004**: 对同一时间窗口和同一事件集合，余额计算结果 MUST 可重复、可审计，避免因刷新次数差异产生不可解释漂移。  

### Key Entities *(include if feature involves data)*

- **AccountSnapshot（账户快照）**: 表示某时刻的最新存款状态；关键属性包括当前余额、初始存款、更新时间。  
- **CashflowEvent（资金事件）**: 表示一次性或周期性入/出项；关键属性包括事件ID、类型（一次性/周期性）、方向（入/出）、金额、生效时间、周期规则、状态。  
- **RealtimeBalanceTick（实时余额刻）**: 表示某一秒用于展示的计算结果；关键属性包括时刻、展示余额、计算来源摘要。  

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 90% 以上用户可在 2 分钟内完成初始化存款设置并在主页看到最新存款金额。  
- **SC-002**: 在包含至少 1 条周期性事件的场景下，主页余额显示在连续 5 分钟观察窗口内每秒均发生按规则更新（允许数值变化为 0 的秒仅在净变化确为 0 时出现）。  
- **SC-003**: 对随机抽样的 100 组事件数据，系统实时显示余额与离线复算结果一致率达到 100%。  
- **SC-004**: 95% 的资金事件新增/修改操作可在 2 秒内反映到主页最新存款显示。  
- **SC-005**: 在内部可用性验证中，至少 20 名参与者完成“余额变化可理解性”5 分量表问卷（1=非常不清晰，5=非常清晰），评分为 4 或 5 的比例 MUST >= 85%，并在 `quickstart.md` 记录样本数、评分分布与统计结果。
